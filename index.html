Persistência de Dados

SQL — A Língua Universal dos Dados

SQL existe desde 1974. Todo banco relacional fala SQL. Oracle, PostgreSQL, MySQL, SQLite, SQL Server — a sintaxe muda em detalhes, o modelo mental é o mesmo. Quem entende SQL de verdade nunca fica preso a um banco específico.

"SQL não é só uma linguagem de consulta. É um modelo mental de como dados se relacionam, se agrupam e se transformam."

SELECT, FROM, WHERE

O núcleo de tudo. Projeção (quais colunas), origem (qual tabela), filtro (qual condição). 90% das queries do mundo são apenas isso.

→ Reconheça: é só filtrar + projetar um conjunto

JOINs

Como tabelas se relacionam. INNER, LEFT, RIGHT, FULL. O conceito é teoria dos conjuntos — interseção, união, diferença.

→ Reconheça: operações em conjuntos relacionados

Agregação

GROUP BY, COUNT, SUM, AVG, MAX, MIN. Reduzir muitas linhas a uma. O mesmo conceito de reduce() em programação funcional.

→ Reconheça: é um fold/reduce sobre grupos

Índices

A estrutura que transforma O(n) em O(log n). Um índice é essencialmente uma B-tree. Entender quando criar (e quando não criar) é crucial.

→ Reconheça: é uma B-tree sobre uma coluna

Transações (ACID)

Atomicidade, Consistência, Isolamento, Durabilidade. Garante que operações complexas não deixam o banco em estado inconsistente.

→ Reconheça: tudo-ou-nada para múltiplas operações

Normalização

1FN, 2FN, 3FN: como estruturar dados para evitar redundância e anomalias de atualização. O modelo mental de "cada fato uma vez".

→ Reconheça: princípio DRY aplicado a dados

Exemplos em Python (sqlite3 — sem framework)

# A mesma lógica funciona em qualquer banco relacional
import sqlite3

conn = sqlite3.connect(":memory:")
cur = conn.cursor()

# DDL: definir estrutura
cur.execute("""
    CREATE TABLE usuarios (
        id INTEGER PRIMARY KEY,
        nome TEXT NOT NULL,
        email TEXT UNIQUE
    )
""")

cur.execute("""
    CREATE TABLE pedidos (
        id INTEGER PRIMARY KEY,
        usuario_id INTEGER REFERENCES usuarios(id),
        valor REAL,
        criado_em TEXT DEFAULT CURRENT_TIMESTAMP
    )
""")

# DML: manipular dados
cur.execute("INSERT INTO usuarios (nome, email) VALUES (?, ?)",
            ("Ana", "ana@email.com"))

# JOIN: relacionar tabelas
cur.execute("""
    SELECT u.nome, COUNT(p.id) AS total_pedidos, SUM(p.valor) AS receita
    FROM usuarios u
    LEFT JOIN pedidos p ON p.usuario_id = u.id
    GROUP BY u.id
    HAVING total_pedidos > 0
    ORDER BY receita DESC
""")

# Transação explícita
with conn:  # commit automático ou rollback em exceção
    cur.execute("UPDATE pedidos SET valor = valor * 1.1 WHERE id = ?", (1,))

O que praticar

  • Window Functions — ROW_NUMBER, RANK, LAG, LEAD: analytics sem subquery
  • CTEs (WITH) — queries legíveis e reutilizáveis dentro de um SELECT
  • EXPLAIN / EXPLAIN ANALYZE — entender o plano de execução de uma query
  • Índices compostos — quando e como criar para queries específicas
  • Modelagem ER — desenhar o schema antes de escrever uma linha de SQL

Comunicação na Web

HTTP — Como a Web Funciona Desde 1991

Toda aplicação web é uma troca de mensagens HTTP. Flask, FastAPI, Django, Express, Rails — todos são apenas abstrações sobre o mesmo protocolo. Entender HTTP diretamente remove a magia de qualquer framework.

ConceitoO que éPor que importa
Métodos (Verbos)
GET POST PUT DELETE PATCH		 A intenção da requisição. GET lê, POST cria, PUT substitui, PATCH atualiza parcialmente, DELETE remove. APIs REST são só essa convenção. Toda API nova que você ver usa esses verbos.
Status Codes
2xx 3xx 4xx 5xx		 200 OK, 201 Created, 301 Redirect, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 500 Server Error. Quando algo falha, o status code diz onde está o problema: no cliente ou no servidor.
Headers Metadados da requisição/resposta. Content-Type, Authorization, Cache-Control, Cookie, Accept. Autenticação, cache, CORS — tudo passa por headers. Ler os headers de uma requisição resolve 80% dos bugs de integração.
Body O conteúdo da mensagem. JSON, form-data, multipart (arquivos), texto puro. É onde os dados vão. Content-Type define como interpretar.
URL / Query String /users/42/orders?status=paid&page=2 — path params + query params. Path identifica o recurso. Query filtra/pagina. Convenção universal.
1991
HTTP/0.9 Só GET, só texto. Uma linha de request, uma resposta.
1996
HTTP/1.0 Headers, status codes, POST. O modelo que ainda usamos conceitualmente.
1997
HTTP/1.1 Keep-Alive (conexão persistente), chunked transfer, virtual hosting.
2015
HTTP/2 Multiplexing, header compression, server push. Mais rápido, mesma semântica.
2022
HTTP/3 (QUIC) UDP em vez de TCP. Mais rápido em redes ruins. O modelo continua o mesmo. 
# HTTP puro com Python — sem framework
from http.server import BaseHTTPRequestHandler, HTTPServer
import json

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/health":
            self._respond(200, {"status": "ok"})
        else:
            self._respond(404, {"error": "not found"})

    def do_POST(self):
        length = int(self.headers["Content-Length"])
        body = json.loads(self.rfile.read(length))
        # processar body...
        self._respond(201, {"created": True})

    def _respond(self, status, data):
        payload = json.dumps(data).encode()
        self.send_response(status)
        self.send_header("Content-Type", "application/json")
        self.send_header("Content-Length", len(payload))
        self.end_headers()
        self.wfile.write(payload)

# Flask, FastAPI, Django — são isso acima com muito açúcar sintático
HTTPServer(("", 8080), Handler).serve_forever()
  • Implementar um servidor HTTP do zero — entender o ciclo request/response sem abstração
  • Usar curl / httpie para debugar APIs sem código
  • Estudar CORS — por que existe, como funciona, como resolver
  • Entender HTTPS / TLS — o handshake, certificados, por que HTTP puro é inseguro
  • Cache HTTP — ETag, Cache-Control, 304 Not Modified

Fundamentos

Estruturas de Dados — Os Blocos de Construção

Linguagem muda, estrutura não. Uma hash map em Python se chama dict, em Java HashMap, em Go map, em Rust HashMap. O comportamento — O(1) médio para insert/lookup — é idêntico. Conhecer a estrutura é conhecer todas as implementações.

Array / Lista

Memória contígua. Acesso por índice O(1). Inserção no meio O(n). A estrutura mais fundamental.

→ Use quando: acesso por posição, iterar sequencialmente

Hash Map / Dict

Chave → valor. Insert/lookup/delete em O(1) médio. Internamente: array + função hash + tratamento de colisão.

→ Use quando: lookup por chave, contar, agrupar

Árvore Binária / BST

Hierarquia. Busca O(log n) quando balanceada. Base dos índices de banco de dados (B-tree), DNS, sistemas de arquivos.

→ Reconheça em: índices SQL, Git commits, JSON

Fila (Queue) e Pilha (Stack)

FIFO e LIFO. Filas aparecem em mensageria (Kafka, RabbitMQ). Pilhas aparecem em call stacks, undo/redo, parsing.

→ Reconheça em: toda fila de mensagens, backtrace de erro

Grafo

Nós + arestas. Redes sociais, mapas, dependências de pacotes, fluxo de trabalho. BFS e DFS são as travessias universais.

→ Reconheça em: npm/pip deps, rotas, recomendações

Heap / Priority Queue

O menor (ou maior) elemento sempre acessível em O(1). Insert/remove em O(log n). Base de algoritmos de agendamento.

→ Reconheça em: Dijkstra, agendadores, top-K queries
# Reconhecendo estruturas por comportamento, não por nome

# Hash Map — O(1) lookup
cache = {}
cache["user:42"] = {"nome": "Ana"}          # insert O(1)
user = cache.get("user:42")               # lookup O(1)

# Pilha — LIFO
historico = []
historico.append("pagina_a")              # push
anterior = historico.pop()                 # pop

# Fila — FIFO (use deque, não list)
from collections import deque
fila = deque()
fila.append("tarefa_1")                   # enqueue O(1)
proxima = fila.popleft()                  # dequeue O(1)

# Heap — menor elemento sempre no topo
import heapq
tarefas = [(3, "baixa"), (1, "critica"), (2, "media")]
heapq.heapify(tarefas)
urgente = heapq.heappop(tarefas)          # (1, "critica")

# Grafo — BFS para caminho mais curto (não ponderado)
from collections import defaultdict, deque

grafo = defaultdict(list)
grafo["A"].extend(["B", "C"])

def bfs(inicio, fim):
    fila = deque([(inicio, [inicio])])
    visitados = {inicio}
    while fila:
        no, caminho = fila.popleft()
        if no == fim: return caminho
        for vizinho in grafo[no]:
            if vizinho not in visitados:
                visitados.add(vizinho)
                fila.append((vizinho, caminho + [vizinho]))

Complexidade e Raciocínio

Algoritmos — Pensar em Custo

Algoritmos não são para decorar — são para desenvolver o instinto de custo. "Isso escala?" é uma pergunta de algoritmo. Big O é a linguagem universal para responder essa pergunta.

ComplexidadeNomeExemplo10x mais dados
O(1)ConstanteDict lookup, array por índiceMesmo tempo
O(log n)LogarítmicaBusca binária, índice B-tree+1 operação
O(n)LinearLoop simples, busca em lista10x mais lento
O(n log n)LinearítmicaMerge sort, quicksort~33x mais lento
O(n²)QuadráticaLoop aninhado, bubble sort100x mais lento
O(2ⁿ)ExponencialForça bruta combinatóriaNão escala

Padrões de Algoritmos (não algoritmos específicos)

Dois Ponteiros

Dois índices percorrendo a mesma estrutura. Reduz O(n²) para O(n) em problemas de par/subarray.

→ Aparece em: validação de palíndromo, soma de dois

Janela Deslizante

Subarray de tamanho fixo ou variável que "desliza". Evita recomputar do zero a cada posição.

→ Aparece em: médias móveis, substring máxima

Dividir para Conquistar

Quebrar problema em subproblemas menores, resolver independentemente, combinar. Merge sort é o exemplo canônico.

→ Aparece em: sort, busca binária, MapReduce

Programação Dinâmica

Memorizar resultados de subproblemas sobrepostos. Transforma exponencial em polinomial.

→ Aparece em: diff de arquivos (git diff), spell check

Greedy (Guloso)

Tomar a melhor escolha local esperando atingir o ótimo global. Simples, nem sempre correto.

→ Aparece em: Huffman encoding, agendamento

Backtracking

Explorar possibilidades, desfazer quando errado. Árvore de decisão com poda.

→ Aparece em: parsers, solvers, geração de permutações

Controle de Versão

Git — O Modelo Mental por Baixo dos Comandos

Git não é uma ferramenta — é um grafo de snapshots imutáveis. Quem entende o modelo (DAG de commits) nunca mais fica perdido. Os comandos são só operações nesse grafo.

"Git armazena snapshots, não diffs. Cada commit é uma foto completa do projeto, com um ponteiro para o commit anterior."

# O modelo mental: Git é um grafo direcionado acíclico (DAG)
#
# main:    A ← B ← C ← D
#                   ↑
# feature:          E ← F
#
# HEAD = onde você está agora
# branch = um ponteiro móvel para um commit
# tag = um ponteiro fixo para um commit

# ─── Operações no grafo ───

# commit: cria novo nó no grafo
git add arquivo.py
git commit -m "feat: adiciona validação de email"

# branch: cria um ponteiro novo
git switch -c feature/login

# merge: une dois ramos do grafo
git merge feature/login          # cria merge commit (preserva histórico)
git merge --squash feature/login # achata em um commit (histórico limpo)

# rebase: reaplica commits sobre outra base
git rebase main                  # lineariza o histórico

# reset: move HEAD para outro commit
git reset --soft HEAD~1   # desfaz commit, mantém mudanças staged
git reset --hard HEAD~1   # desfaz commit E as mudanças

# cherry-pick: copia um commit específico
git cherry-pick abc1234   # traz apenas esse commit para a branch atual

# bisect: busca binária para encontrar bug
git bisect start
git bisect bad            # commit atual tem o bug
git bisect good v1.0      # essa versão estava ok
# git faz checkout automático no meio → você testa → repete
  • Entender o .git/ — inspecionar objects/, refs/, HEAD direto no disco
  • Conventional Commits — feat:, fix:, chore:: padrão usado em todo lugar
  • git reflog — recuperar commits "perdidos" após reset --hard
  • git stash — guardar trabalho incompleto temporariamente
  • Estratégias de branching — GitFlow vs trunk-based: entender o trade-off

Diagnóstico

Debugging & Logs — O Problema Sempre Está Ali

A habilidade mais transferível de todas. A linguagem muda, o processo de debugar não. Quem sabe ler logs e usar um debugger bem nunca fica preso por muito tempo em nenhuma stack.

Leitura de Stack Trace

Leia de baixo para cima. A última linha é onde explodiu. O frame mais alto com seu código é onde investigar.

→ Padrão universal: toda linguagem tem stack trace

Logging Estruturado

JSON logs com campos consistentes (timestamp, level, message, context). Muito mais útil do que print() aleatório.

→ Padrão universal: Splunk, ELK, CloudWatch — todos esperam isso

Bisect Mental

Isolar o problema pela metade: "funciona até aqui, quebra depois daqui". Busca binária no código.

→ Mesma lógica do git bisect aplicada no raciocínio

Reproduzir Minimamente

Criar o menor caso possível que reproduz o bug. Isso frequentemente revela a causa sozinho.

→ Habilidade central para abrir boas issues no GitHub
# Logging estruturado em Python — o padrão de produção
import logging
import json
from datetime import datetime

class JSONFormatter(logging.Formatter):
    def format(self, record):
        return json.dumps({
            "ts": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "msg": record.getMessage(),
            "module": record.module,
            **getattr(record, "extra", {}),
        })

logger = logging.getLogger("app")
handler = logging.StreamHandler()
handler.setFormatter(JSONFormatter())
logger.addHandler(handler)

# Logging com contexto — indispensável em produção
def processar_pedido(pedido_id, usuario_id):
    log = logging.LoggerAdapter(logger, {
        "pedido_id": pedido_id,
        "usuario_id": usuario_id,
    })
    log.info("Processando pedido")
    try:
        # lógica de negócio...
        log.info("Pedido aprovado")
    except Exception as e:
        log.error("Falha ao processar pedido", exc_info=True)
        raise

Arquitetura

Padrões de Design — Soluções com Nome

Padrões de design são soluções nomeadas para problemas recorrentes. Quando você vê um Decorator no Python, um Middleware no Express, ou um Interceptor no Spring — todos são o mesmo padrão com nomes diferentes.

Repository

Abstrai o acesso a dados. Sua lógica de negócio não sabe se os dados vêm de SQL, Redis ou API.

→ Aparece em: todo ORM, Clean Architecture, DDD

Observer / Event

Publicar um evento; múltiplos ouvintes reagem independentemente. Desacopla quem produz de quem consome.

→ Aparece em: DOM events, Kafka, webhooks, React state

Decorator / Middleware

Envolver uma função com comportamento extra sem modificá-la. Pilha de responsabilidades.

→ Aparece em: @decorator Python, Express middleware, logging

Factory

Separar a criação de objetos do seu uso. O código que usa não sabe como foi criado.

→ Aparece em: DI containers, conexões de banco, drivers

Strategy

Encapsular algoritmos intercambiáveis. Trocar o comportamento em runtime sem mudar o código cliente.

→ Aparece em: ordenação customizada, autenticação, pagamento

Circuit Breaker

Parar de chamar um serviço que está falhando. Evita cascade failure em sistemas distribuídos.

→ Aparece em: Hystrix, Resilience4j, qualquer microserviço
# Repository Pattern — independente de banco
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class Usuario:
    id: int
    nome: str
    email: str

class UsuarioRepository(ABC):  # contrato
    @abstractmethod
    def buscar_por_id(self, id: int) -> Usuario | None: ...
    @abstractmethod
    def salvar(self, usuario: Usuario) -> None: ...

class UsuarioRepositorySQL(UsuarioRepository):  # implementação SQL
    def buscar_por_id(self, id): ...
    def salvar(self, usuario): ...

class UsuarioRepositoryMemoria(UsuarioRepository):  # implementação para testes
    def __init__(self): self._dados = {}
    def buscar_por_id(self, id): return self._dados.get(id)
    def salvar(self, u): self._dados[u.id] = u

# Decorator Pattern — middleware
import time
from functools import wraps

def medir_tempo(fn):
    @wraps(fn)
    def wrapper(*args, **kwargs):
        inicio = time.perf_counter()
        resultado = fn(*args, **kwargs)
        duracao = time.perf_counter() - inicio
        print(f"{fn.__name__} levou {duracao:.3f}s")
        return resultado
    return wrapper

Execução Paralela

Concorrência — O Mesmo Problema em Toda Linguagem

Threads, processos, coroutines, fibers — diferentes mecanismos, mesmo problema: como executar múltiplas coisas ao mesmo tempo sem corromper o estado compartilhado. Os conceitos de race condition, deadlock e mutex existem em toda linguagem.

Callbacks
Node.js clássico, C assíncrono Passa uma função para ser chamada quando pronto. Callback hell em operações aninhadas.
Promises
JavaScript ES6, Python Futures Objeto que representa um valor futuro. Composição com .then() / .catch().
Async/Await
Python asyncio, JS async, Rust async, Go goroutines Açúcar sintático sobre Promises/coroutines. Código assíncrono parece síncrono.
Actors / CSP
Erlang/Elixir Actors, Go channels Comunicação por mensagens em vez de memória compartilhada. Evita locks. 
# Os três modelos de concorrência em Python
import asyncio
import threading
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# ── 1. I/O Bound → asyncio (uma thread, cooperativo)
async def buscar_dados(url: str) -> dict:
    async with aiohttp.ClientSession() as s:
        async with s.get(url) as r:
            return await r.json()

async def buscar_todos(urls):
    return await asyncio.gather(*[buscar_dados(u) for u in urls])

# ── 2. I/O Bound Bloqueante → ThreadPool
def query_banco(id): ...  # operação bloqueante

with ThreadPoolExecutor(max_workers=10) as pool:
    resultados = list(pool.map(query_banco, ids))

# ── 3. CPU Bound → ProcessPool (contorna o GIL)
def calcular(n): return sum(range(n))

with ProcessPoolExecutor() as pool:
    resultados = list(pool.map(calcular, [10**6]*4))

Segurança

Segurança — Os Ataques Que Não Mudam

XSS, SQL Injection, CSRF existem desde os anos 90 e ainda derrubam sistemas modernos. As ferramentas mudam (frameworks ajudam a prevenir), mas o engenheiro que entende o ataque nunca é pego de surpresa.

SQL Injection

Input do usuário interpretado como SQL. Prevenção: sempre use parâmetros (?), nunca concatenação de string.

→ Princípio: separar dados de código

XSS (Cross-Site Scripting)

Input do usuário renderizado como HTML/JS. Prevenção: escapar output, Content Security Policy.

→ Princípio: nunca confiar em input, sempre escapar output

CSRF

Requisição forjada de outro site. Prevenção: tokens CSRF, SameSite cookies, verificar Origin.

→ Princípio: verificar que a requisição veio de onde parece

Senhas — Hashing

Nunca armazenar senha em texto puro. Sempre usar bcrypt, scrypt ou Argon2 (não MD5, não SHA1).

→ Princípio: guardar a prova, não o segredo

Principle of Least Privilege

Cada componente acessa apenas o que precisa. O banco de dados de leitura não precisa de permissão de escrita.

→ Princípio universal: minimizar superfície de ataque

Secrets Management

Nunca commitar segredos no Git. Variáveis de ambiente, Vault, Secret Manager — o padrão é o mesmo.

→ Princípio: segredos fora do código

Não é para estudar tudo de uma vez. É um mapa: você sabe onde está, sabe para onde ir, e cada coisa aprendida se conecta com as outras.
